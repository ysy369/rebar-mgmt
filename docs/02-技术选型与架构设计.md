## 2. 技术选型说明

### 2.1 后端：Python 3 + Flask

| 库 | 版本 | 用途 |
|----|------|------|
| Flask | 3.x | Web 框架，轻量灵活 |
| Flask-SQLAlchemy | 3.x | ORM 数据库操作 |
| Flask-Migrate | 4.x | 数据库迁移（Alembic） |
| Flask-Login | 0.6+ | 用户会话管理 |
| bcrypt | 4.x | 密码哈希 |
| pandas | 2.x | Excel 数据解析与处理 |
| openpyxl | 3.x | Excel 读写（保留公式/超链接） |
| gunicorn | 21.x | 生产环境 WSGI 服务器 |

> **选择理由**：语法简单，运维人员学习曲线低；openpyxl/pandas 对 Excel 支持成熟；Flask 轻量可渐进扩展。

### 2.2 数据库：MySQL 8.0

- **引擎**：InnoDB（支持事务、外键）
- **字符集**：utf8mb4
- **时区**：Asia/Shanghai

> **选择理由**：Windows 运行稳定，运维团队最熟悉，备份恢复生态完善。

### 2.3 前端：Bootstrap 5 + Chart.js

| 库 | 用途 | 加载方式 |
|----|------|----------|
| Bootstrap 5 | 布局、组件、响应式 | CDN |
| Bootstrap Icons | 图标 | CDN |
| Chart.js | 趋势图、柱状图 | CDN |

> **选择理由**：无需 Node.js / npm 构建工具，CDN 引入即可使用，减少运维负担。

### 2.4 Excel 处理：openpyxl + pandas

| 库 | 用途 |
|----|------|
| pandas | 读取 Excel → DataFrame → 数据清洗 → 批量入库 |
| openpyxl | 写 Excel → 保留公式、超链接、样式 |

> **选择理由**：pandas 处理表格数据高效；openpyxl 是唯一支持 .xlsx 公式写入的纯 Python 库。

## 3. 架构设计

### 3.1 整体架构：分层单体

```
┌──────────────────────────────────────┐
│          Templates (Jinja2)          │  ← 视图层
├──────────────────────────────────────┤
│          Routes (Blueprint)          │  ← 路由层
├──────────────────────────────────────┤
│          Services (业务逻辑)          │  ← 服务层
├──────────────────────────────────────┤
│          Models (SQLAlchemy)         │  ← 数据层
└──────────────────────────────────────┘
```

### 3.2 路由设计（Blueprint）

| Blueprint | URL 前缀 | 说明 |
|-----------|----------|------|
| `auth_bp` | `/auth` | 登录、登出、密码修改 |
| `admin_bp` | `/admin` | 管理端（需 admin 角色） |
| `project_bp` | `/project` | 项目端（需登录） |

### 3.3 权限验证流程

```
用户请求 → Flask-Login 验证登录状态
           ├── 未登录 → 跳转 /auth/login
           └── 已登录 → 检查路由权限
                        ├── /admin/* → 验证 role == 'admin'
                        └── /project/* → 验证项目授权
```

### 3.4 数据流

```
[Excel 上传]
    │
    ▼
[文件保存到 uploads/]
    │
    ▼
[pandas 读取 + 校验]
    │
    ├── 校验失败 → 返回错误详情
    │
    └── 校验成功
         │
         ▼
    [数据写入数据库（事务）]
         │
         ▼
    [触发汇总计算]
         │
         ▼
    [返回导入成功 + 汇总结果]
```

## 4. 部署架构

### 4.1 开发环境
```bash
# 直接启动（本地开发调试）
python run.py → Flask 内置服务器 (127.0.0.1:5000)

# 或使用 Docker Compose 启动（与生产一致）
docker compose up -d
```

### 4.2 生产环境：Docker Compose（主方案）

```
┌─────────────────────────────────────────────┐
│              Windows Server                  │
│  ┌───────────────┐  ┌────────────────────┐  │
│  │ rebar-app      │  │ rebar-db           │  │
│  │ (Flask+waitress)│  │ (MySQL 8.0)        │  │
│  │ port 5000      │  │ port 3306 (internal)│  │
│  └───────────────┘  └────────────────────┘  │
│         ↑                    ↑              │
│         └──── Docker Network ┘              │
│                                              │
│  用户浏览器 → http://服务器IP:5000             │
└─────────────────────────────────────────────┘
```

- **无需在宿主机安装 Python 或 MySQL**
- 一条 `docker compose up -d` 即可启动
- 数据卷挂载到宿主机，数据持久化
- MySQL 端口不对外暴露（仅容器间通信）

### 4.3 容器化方案详述

采用 Docker Compose 双容器架构：

| 容器 | 镜像 | 说明 |
|------|------|------|
| `rebar-app` | `python:3.11-slim` + Flask | 应用服务，对外开放 5000 端口 |
| `rebar-db` | `mysql:8.0` | 数据库，仅容器网络内可访问 |

关键配置：
- 应用代码通过 **bind mount** 挂载（开发时可热更新）
- MySQL 数据通过 **named volume** 持久化到宿主机
- 应用通过 `depends_on` 等待数据库就绪后才启动
- 数据库初始化脚本放在 `docker-entrypoint-initdb.d/` 自动执行

### 4.4 备用方案：Windows 原生部署
```powershell
# 如果有 Python 环境，也可直接注册为 Windows Service
nssm install RebarMgmt "C:\Python39\python.exe" "C:\rebar-mgmt\run.py"
nssm set RebarMgmt AppDirectory "C:\rebar-mgmt"
nssm start RebarMgmt
```

## 5. 安全设计

| 层面 | 措施 |
|------|------|
| 密码 | bcrypt 哈希，迭代 12 轮 |
| SQL | SQLAlchemy ORM 参数化查询，防注入 |
| XSS | Jinja2 自动转义 + 输入校验 |
| 文件上传 | 仅允许 .xlsx，限制 10MB，校验文件头 |
| Session | HttpOnly Cookie，8 小时超时 |
| 权限 | 服务端验证角色，每个请求检查项目授权 |
| CSRF | Flask-WTF CSRF Token（如使用 WTForms） |
